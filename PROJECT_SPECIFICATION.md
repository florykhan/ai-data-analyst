# AI DataLab — Project Idea & Master Plan

**One-sentence pitch:** A Databricks-style, AI-first data analytics platform (smaller in scope) where users upload data, explore it with natural language, train ML models, and deploy predictions—built with AWS, SQL, TypeScript, Java, and an AI agent.

---

## Part 1: The Big Idea

### Vision

**AI DataLab** is your flagship project: a **unified data analytics and ML platform** that feels like a lighter, more accessible **Databricks**. It brings together:

- **Data ingestion and storage** (like a small data lake)
- **SQL and programmatic access** to that data
- **An AI agent** that understands your data and turns natural language into queries, visualizations, and model training
- **Machine learning** from exploratory modeling to trained models and prediction APIs
- **Cloud-native infrastructure** on **AWS**, with clear use of **SQL** (PostgreSQL/RDS, and SQL generated by the AI), **TypeScript** (frontend and possibly BFF/Node services), and **Java** (robust backend services, data pipelines, or ML serving)

The big idea is: **anyone with a spreadsheet or a CSV can become a data analyst.** They don’t need to learn SQL, Python, or ML tooling first. They talk to an AI agent, upload data, click through auto-generated insights, train models with guidance, and ship predictions via API—all in one place.

### Why “Databricks-like but smaller”

- **Databricks** = data lake + notebooks + Spark + ML + collaboration + enterprise scale.  
- **We** = curated feature set: upload → profile → query (SQL + NL) → visualize → train ML → deploy API, with an AI agent as the main interface. Same *kind* of value (unified analytics + ML), smaller surface area and scope.

So the “big idea” is: **one platform where data lands, gets profiled, queried (by SQL or by talking to an AI), visualized, modeled, and served—with AWS, SQL, TypeScript, and Java as core tech.**

### Core principles

1. **AI-first:** The primary way to “do” analysis is through the AI agent (chat + suggestions), with SQL and forms as power-user options.
2. **Data-centric:** Everything is organized around datasets and their metadata (stored and queried with SQL).
3. **ML as a product:** Training, evaluation, and prediction APIs are first-class; we implement real pipelines, not demos.
4. **Cloud-native:** Design for AWS from the start (compute, storage, DB, monitoring).
5. **Polyglot by design:** Use the right tool per layer—TypeScript for UI and possibly API layer, Java for heavy lifting or services, Python for ML, SQL for data and metadata.

## Part 2: What We’ll Implement

Below is the full scope: the “what” of the project. The “how” and “when” come in the step-by-step plan.

### 2.1 Data ingestion & storage

- **Upload formats:** CSV, JSON, Excel (and optionally Parquet later).
- **Validation:** File type, size limits, basic schema checks.
- **Storage:** 
  - Raw and processed assets in **AWS S3** (buckets for uploads, processed datasets, model artifacts).
  - Metadata (datasets, jobs, users, models) in a **SQL database** (e.g. **PostgreSQL on RDS**).
- **Ingestion pipeline:** 
  - Optional **Java** service for high-throughput or validated ingestion (e.g. validate → write to S3 → write metadata via SQL).
  - Or Python/FastAPI for v1; Java can be introduced in a later phase.
- **Data catalog:** Tables in SQL describing datasets (name, schema, row count, column types, S3 paths, created_at, etc.) so the AI and UI can list and query “what’s available.”

### 2.2 Data profiling & discovery

- **Automatic profiling** after upload:
  - Column types (numeric, categorical, datetime, text).
  - Missing values, distinct counts, min/max/mean where applicable.
  - Simple distribution stats (e.g. histograms bins), correlations for numeric columns.
- **Stored in SQL:** Profile results (or references to S3 reports) so the AI agent and dashboards can answer “what’s in this dataset?” without re-scanning.
- **Discovery:** List datasets, filter by name/date/size; expose via API and to the AI agent.

### 2.3 SQL layer

- **Structured querying:** 
  - Either run **SQL against a mirror of the data** (e.g. tables in PostgreSQL or a query engine on top of S3), or
  - Generate and run **SQL on an in-process/store copy** (e.g. DuckDB, or PostgreSQL with loaded tables).
- **AI-generated SQL:** The AI agent produces **SQL** from natural language; we validate (read-only where possible) and execute, then return results.
- **Audit:** Log queries (and optionally results size) in SQL for debugging and cost control.

### 2.4 AI agent

- **Conversational interface:** Chat where the user refers to “this dataset” or “sales_data” and asks questions.
- **Capabilities:**
  - **SQL generation:** “Top 5 customers by revenue” → generated SQL → execution → return table or chart.
  - **Data interpretation:** “What’s in this dataset?”, “Which columns are numeric?” → use catalog + profile from SQL.
  - **Visualization requests:** “Plot revenue by month” → agent decides chart type + SQL → return chart spec or image.
  - **ML suggestions:** “I want to predict churn” → suggest target column, model type, and kick off training.
- **Context:** Agent gets dataset schema, sample stats, and recent conversation; optionally a **RAG** over docs or past queries.
- **Implementation:** Backed by an LLM (e.g. OpenAI/Anthropic); **TypeScript** or **Java** can call the LLM and orchestrate (query generation → execution → response formatting).

### 2.5 Visualizations & dashboards

- **Auto-charts:** After profiling, auto-generate:
  - Histograms, bar charts, correlation heatmaps, time series, scatter plots.
- **On-demand charts:** From AI agent or from “build chart” UI (select dataset, columns, chart type).
- **Dashboard:** One or more views per dataset (overview, distributions, correlations, model performance); all interactive (e.g. **Plotly** or **Recharts** in **TypeScript** frontend).
- **Export:** PNG/CSV export for charts and tables.

### 2.6 Machine learning pipeline

- **Model types:** Regression, classification, clustering (as in README).
- **Algorithms:** Linear regression, Random Forest, Gradient Boosting, Logistic Regression, K-Means (and optionally XGBoost).
- **Pipeline steps:** 
  - Load data from S3 or SQL.
  - Preprocessing: missing value handling, categorical encoding, normalization.
  - Train/test split; training; evaluation (RMSE, accuracy, F1, ROC-AUC, etc.).
- **Feature importance:** Expose to UI and to AI agent for “what drives the outcome?”.
- **Artifacts:** Save model binaries (e.g. to S3); metadata (algorithm, hyperparams, metrics) in **SQL**.
- **Implementation:** Python (scikit-learn, XGBoost) is natural; **Java** can be used for model serving (e.g. REST API that loads serialized model and predicts) or for orchestration if we want a polyglot stack.

### 2.7 Prediction API

- **Deploy:** “Deploy model” creates an API (e.g. POST `/models/{id}/predict`) that accepts feature vector or JSON and returns prediction(s).
- **Auth & rate limiting:** API key or JWT; optional rate limits (e.g. per API key).
- **Implementation:** Can be **Java** (e.g. Spring Boot) or Python (FastAPI); both can load models from S3 and read metadata from SQL.

### 2.8 User & auth (optional for v1)

- **Auth:** Login/signup (e.g. JWT); optional OAuth (Google/GitHub).
- **Multi-tenancy:** Datasets and models scoped by user/org; all metadata in **SQL** with `user_id` or `org_id`.
- **RBAC (later):** Roles (viewer, analyst, admin) for shared workspaces.

### 2.9 Infrastructure & DevOps

- **AWS:**
  - **S3:** Raw data, processed data, model artifacts.
  - **RDS (PostgreSQL):** Metadata, catalog, profiles, users, jobs.
  - **Compute:** EC2 or ECS for backend(s); Lambda for small, event-driven tasks if useful.
  - **API Gateway + ALB:** Single entry for APIs.
  - **CloudWatch:** Logs, metrics, alarms.
- **SQL:** All relational state in PostgreSQL; migrations (e.g. Flyway for **Java**, Alembic for Python).
- **IaC:** Terraform or CDK for AWS (in `infrastructure/aws/`).
- **Containers:** Docker for backend(s); optional ECR + ECS.
- **CI/CD:** Build, test, deploy backend and frontend (e.g. GitHub Actions).

### 2.10 Frontend (TypeScript)

- **Stack:** **TypeScript** + React (or Next.js) + TailwindCSS; Plotly/Recharts for charts.
- **Screens:**
  - Login (if auth).
  - Dataset list + upload.
  - Dataset detail: profile, sample rows, auto-visualizations.
  - Chat interface (AI agent).
  - “Run SQL” or “Query” (optional power user).
  - Model training: select dataset, target, type, algorithm → run → view metrics and feature importance.
  - Model list + deploy → prediction API docs/key.
  - Dashboard (overview of datasets + models).
- **State:** React state or React Query; API client in TypeScript calling backend.

## Part 3: Technology Stack (AWS, SQL, TypeScript, Java)

### 3.1 AWS

- **S3:** Datasets (raw/processed), model binaries, optional static frontend.
- **RDS (PostgreSQL):** Single source of truth for metadata (datasets, profiles, models, users, jobs, audit logs).
- **EC2 or ECS:** Run backend services (Java and/or Python).
- **VPC, security groups, IAM:** Restrict access to RDS and S3; least privilege for app roles.
- **CloudWatch:** Logs, metrics (request count, latency, errors); alarms for RDS CPU, S3 errors, etc.
- **Optional:** Lambda for event-driven triggers; API Gateway if we want managed API layer.

### 3.2 SQL

- **PostgreSQL:** 
  - Tables: `users`, `datasets`, `dataset_columns`, `profiles`, `models`, `model_runs`, `predictions_audit`, `chat_sessions`, `query_logs`, etc.
  - Use **SQL** for all reads/writes of this metadata; the AI agent can “see” schema and sample stats from SQL.
- **SQL generation by AI:** Agent outputs SQL that we run (read-only) against a safe schema (e.g. dataset-specific views or a dedicated query DB) so we don’t corrupt catalog data.
- **Migrations:** Versioned schema changes (Flyway with **Java**, or Alembic with Python).

### 3.3 TypeScript

- **Frontend:** Entire UI in **TypeScript** (React or Next.js): upload, dashboards, chat, model training, settings.
- **Optional:** Node.js/TypeScript BFF (Backend-for-Frontend) that calls Java/Python and LLM, and shapes responses for the UI.
- **Shared types:** Define DTOs (dataset, model, profile, chat message) in TypeScript and keep in sync with backend contracts (OpenAPI → TS client if possible).

### 3.4 Java

- **Use cases:**
  - **Data ingestion service:** Validate uploads, write to S3, insert metadata into PostgreSQL (high throughput, strong typing).
  - **Prediction API:** Load model from S3 (or from DB reference), run inference (via PMML/ONNX or calling Python microservice), return JSON; auth and rate limiting in Java.
  - **Orchestration:** Job queue (e.g. SQS) consumer that triggers profiling or ML training (could call Python worker).
- **Stack:** Spring Boot (REST, security, DB with JPA/Hibernate or jdbc); Maven or Gradle; **SQL** via JPA or raw JDBC.

### 3.5 Python

- **ML pipeline:** scikit-learn, pandas, XGBoost; preprocessing, training, evaluation, feature importance.
- **FastAPI (optional):** Dataset processing, profiling, chat orchestration (call LLM, run SQL), or we move some of this to Java over time.
- **Scripts:** One-off EDA or model export (e.g. to ONNX) for **Java** serving.

### 3.6 AI / LLM

- **Provider:** OpenAI or Anthropic (or both, behind an abstraction).
- **Roles:** Generate SQL, interpret data, suggest visualizations, suggest ML steps; optionally summarize long outputs for chat.
- **Safety:** Validate generated SQL (read-only, no DDL); sanitize inputs; rate limit and cost caps.

## Part 4: AI Agent & Machine Learning (Deeper)

### 4.1 AI agent design

- **Inputs:** User message + current dataset context (name, columns, types, sample stats) + optional conversation history.
- **Outputs:** 
  - Structured actions: `run_sql`, `show_profile`, `create_chart`, `suggest_model`, `start_training`.
  - Or direct answer (e.g. “This dataset has 10 columns; the target is numeric.”).
- **Flow:** User asks → backend sends prompt to LLM → LLM returns action + params → backend executes (SQL, profile lookup, train job) → result formatted and sent to frontend (table, chart, or text).
- **Tools (function calling):** Expose “run_sql”, “get_dataset_profile”, “list_datasets”, “train_model”, “create_chart” as tools so the LLM can choose and parameterize them.

### 4.2 Machine learning implementation

- **Training:** Triggered by UI or by AI agent; runs in a worker (Python) that reads from S3/SQL, trains, writes model to S3 and metrics to SQL.
- **Evaluation:** Always compute relevant metrics (RMSE, accuracy, F1, etc.); store in `model_runs` (SQL).
- **Feature importance:** Tree-based models and linear models; store and show in UI and to agent.
- **Serving:** Either **Java** service that loads serialized model (PMML/ONNX) or HTTP call to Python microservice; both read model location from SQL and credentials from env/IAM.

## Part 5: Step-by-Step Implementation Plan

A phased plan so the project stays manageable and each phase delivers something usable.

### Phase 0: Foundation (weeks 1–2)

1. **Repo & structure:** Already done (frontend, backend, data, infrastructure, docs).
2. **.gitignore and project-specific ignores:** Done.
3. **SQL schema (PostgreSQL):** Design and implement initial tables:
   - `users` (id, email, name, created_at),
   - `datasets` (id, user_id, name, s3_key, format, row_count, created_at),
   - `dataset_columns` (id, dataset_id, name, dtype, nullable, stats_json),
   - `profiles` (id, dataset_id, profile_json, created_at).
4. **Migrations:** Set up Flyway (Java) or Alembic (Python) and apply schema.
5. **AWS base:** Create VPC, S3 bucket(s), RDS PostgreSQL (or local Postgres for dev); document in `infrastructure/aws/`.
6. **Local dev:** Docker Compose for Postgres + optional local S3 (MinIO); README for “how to run backend and DB.”

### Phase 1: Data ingestion & catalog (weeks 3–5)

7. **Upload API:** Endpoint POST `/datasets/upload` (multipart); validate file type and size; store file in S3; insert row in `datasets` and `dataset_columns` (basic schema from header or first row). Implement in Python (FastAPI) or **Java** (Spring Boot).
8. **List datasets:** GET `/datasets` (and GET `/datasets/{id}`) reading from **SQL**; return to **TypeScript** frontend.
9. **Frontend (TypeScript):** Simple UI: login (optional stub), list of datasets, upload form (drag-and-drop). Use shared types for Dataset.
10. **Profiling job:** After upload, enqueue or run a job that computes profile (column types, nulls, min/max, distinct count); write result to `profiles` and optionally extend `dataset_columns.stats_json`. Store in **SQL**.
11. **GET dataset summary:** GET `/datasets/{id}/summary` returns profile from **SQL** (and optionally sample rows from S3/DB).

### Phase 2: SQL & query layer (weeks 6–8)

12. **Queryable data:** Decide approach: (A) copy ingested data into PostgreSQL (e.g. per-dataset table or schema), or (B) use DuckDB/Athena on S3. Implement one approach.
13. **Run SQL API:** POST `/query` or `/datasets/{id}/query` with `{ "sql": "..." }`; validate read-only; execute; return rows (and optionally chart config). Log query in `query_logs` (SQL).
14. **Frontend:** “Query” or “Run SQL” page: text area for SQL, run button, table (and optional chart) result. **TypeScript** only.
15. **Safety:** Sanitize/validate SQL (no DDL, no multi-statement if risky); timeouts and row limits.

### Phase 3: AI agent (weeks 9–12)

16. **LLM integration:** Backend module that calls OpenAI/Anthropic with system prompt (“You are a data analyst assistant…”) and user message; optional **function/tool** definitions for `run_sql`, `get_profile`, `list_datasets`.
17. **Agent orchestration:** On POST `/chat/query`, build context (dataset schema + profile from **SQL**), call LLM, parse response (e.g. tool call or direct answer), execute tool (run SQL, get profile), format result, optionally call LLM again to summarize. Return to frontend.
18. **Chat UI (TypeScript):** Chat component: message list, input, send; display text, tables, and charts (from agent responses). State in React; API client in TypeScript.
19. **Conversation storage (optional):** Save chat sessions and messages in **SQL** (`chat_sessions`, `chat_messages`) for history and future RAG.
20. **Hardening:** SQL validation before execution; input length limits; rate limiting and cost controls for LLM.

### Phase 4: Visualizations & dashboards (weeks 13–15)

21. **Chart API:** Endpoint that accepts dataset id + chart config (type, x, y, aggregation) and returns data (or frontend fetches data and builds chart). Or agent returns chart spec and frontend fetches data.
22. **Auto-charts:** After profile, generate default chart configs (histogram per numeric column, bar for top categories, correlation matrix). Store or compute on demand; serve to frontend.
23. **Charts in TypeScript:** Use Plotly or Recharts; reusable components in `frontend/charts/`; use in dataset detail page and in chat (inline charts).
24. **Dashboard page:** One page per dataset: overview stats, list of auto-charts, link to chat and to “Train model.” Optional multi-dataset dashboard (tiles per dataset).

### Phase 5: ML pipeline (weeks 16–20)

25. **Model metadata (SQL):** Tables `models` (id, dataset_id, name, type, target_column, algorithm, status, s3_model_key, created_at), `model_runs` (id, model_id, metrics_json, created_at).
26. **Train API:** POST `/models/train` with dataset_id, target, model type (regression/classification/clustering), algorithm. Backend enqueues job or runs async; worker loads data (from S3/SQL), preprocesses, trains, saves model to S3, writes to `models` and `model_runs`. Implement in Python.
27. **Training UI (TypeScript):** Form: select dataset, target column, type, algorithm; “Train” button; poll or WebSocket for status; show metrics and feature importance when done.
28. **List models:** GET `/models` (and GET `/models/{id}`) from **SQL**; show in frontend with status and metrics.
29. **Evaluation display:** Metrics (RMSE, accuracy, F1, etc.) and feature importance in UI; optional confusion matrix or ROC curve for classification.

### Phase 6: Prediction API & serving (weeks 21–23)

30. **Prediction endpoint:** POST `/models/{id}/predict` with JSON body (feature key-value or array). Load model (from S3 or cache); run inference; return predictions. Implement in **Java** (e.g. Spring Boot + PMML/ONNX) or Python (FastAPI + joblib); choose one for v1.
31. **Auth for API:** API key or JWT for prediction endpoint; store keys in **SQL**; validate in middleware.
32. **Frontend:** “Deploy” or “API” tab per model: show endpoint, example request, and API key (if we generate one). **TypeScript** only.
33. **Rate limiting (optional):** Per API key or per user; document in docs.

### Phase 7: Infrastructure & polish (weeks 24–26)

34. **AWS production:** Deploy backend(s) on EC2 or ECS; RDS in private subnet; S3 buckets with proper IAM; CloudWatch dashboards and alarms. Use **Terraform** or CDK in `infrastructure/aws/`.
35. **CI/CD:** GitHub Actions (or similar): build **TypeScript** frontend and **Java**/Python backend; run tests; deploy to staging/production.
36. **Docs:** Update `docs/architecture.md` and `docs/api.md` with final APIs and diagrams; add a short “Run locally” and “Deploy” guide.
37. **Security review:** Auth, SQL injection prevention, file upload validation, secrets in env/Secrets Manager.
38. **Performance:** Indexes on **SQL** tables; caching (e.g. profile, model metadata) where useful; frontend bundle size and lazy loading.

### Phase 8: Extras (backlog)

39. **AutoML:** Auto-select algorithm and hyperparameters; run multiple models and pick best (by metric).
40. **Model monitoring:** Log prediction inputs/outputs; basic drift or accuracy monitoring over time.
41. **Collaboration:** Share dataset or project with another user; permissions in **SQL**.
42. **Streaming (advanced):** Ingest from Kafka/Kinesis; real-time aggregations (stretch goal).
43. **Notebook-style UI (optional):** Code cells that run SQL or Python (sandboxed) for power users; optional **Java** backend for job execution.

## Part 6: Success Criteria & Milestones

- **M1 (Phase 1):** User can upload a CSV, see it in the list, and see basic profile (column types, counts) from **SQL**.
- **M2 (Phase 2):** User can run **SQL** on a dataset (via API and UI) and see results in a table.
- **M3 (Phase 3):** User can ask the AI agent a natural language question and get an answer backed by **SQL** or profile data.
- **M4 (Phase 4):** Each dataset has an overview dashboard with auto-generated charts (TypeScript frontend).
- **M5 (Phase 5):** User can train a regression/classification model from the UI; metrics and feature importance are stored in **SQL** and displayed.
- **M6 (Phase 6):** User can call a **prediction API** (with auth) and get predictions from a trained model.
- **M7 (Phase 7):** Platform runs on **AWS** (S3, RDS, compute); CI/CD and basic monitoring in place.
